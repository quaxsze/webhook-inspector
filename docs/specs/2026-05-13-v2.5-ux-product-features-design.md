# V2.5 — UX Product Features Design

**Date:** 2026-05-13
**Status:** Approved — implementation pending
**Scope:** Three independent UX features bundled as a single release
**Estimated effort:** ~9h total

---

## Goals

Improve the day-to-day usability of webhook-inspector with three features users of comparable tools (webhook.site, requestbin) expect:

1. **Vanity URL slug** — let users pick a memorable URL instead of a random token
2. **Search/filter** — find a specific request among hundreds of captured ones
3. **Export JSON** — download captured requests for offline analysis or replay

Each feature is independently shippable and independently testable. They share a release because their combined surface area is small enough that bundling reduces deployment churn.

## Non-goals

- **No authentication or ownership.** V2.5 trusts the URL-as-secret model. Owner-based slug claiming and revocation lands in V5 with Google OAuth.
- **No filtering on export.** Export dumps everything; combining export with search is deferred.
- **No advanced search operators.** No field-scoping (`method:POST`), no boolean operators, no regex. Plain text only.
- **No anti-squatting protection.** First-come-first-served. Documented as a known limitation.

## Feature 1: Vanity URL slug

### User story

> As a developer testing Stripe webhooks, I want my inspector URL to be `hook.odessa-inspect.org/h/stripe-test` instead of `hook.odessa-inspect.org/h/k7Hq3...`, so I can recognize it in my dashboard tabs.

### Behavior

- `POST /api/endpoints` accepts an optional `slug` field in the request body.
- If `slug` is provided and free → endpoint created with `token = slug`. The slug IS the token.
- If `slug` is provided and taken → `409 Conflict` with message `"slug already taken"`.
- If `slug` is provided and invalid format → `400 Bad Request` with message describing the format constraint.
- If `slug` is omitted → existing behavior (`secrets.token_urlsafe(16)`) preserved.

### Constraints

- **Format regex:** `^[a-z0-9][a-z0-9-]{1,30}[a-z0-9]$` (3-32 chars, lowercase letters + digits + hyphens, no leading/trailing hyphen). Consecutive hyphens are allowed (`foo--bar` is valid) — keeping the regex simple.
- **Reserved slugs** (rejected with `400`):
  `api`, `h`, `health`, `stream`, `static`, `admin`, `app`, `hook`, `www`, `assets`, `docs`, `stripe`, `github`, `slack`, `shopify`, `twilio`, `zapier`, `discord`.
  Stored as a frozenset in `domain/entities/endpoint.py`.

### Data model

**No new column.** The slug reuses the existing `token` column on `endpoints` (already `UNIQUE NOT NULL` via `ix_endpoints_token`). The validation runs in the application layer **only when the slug is user-provided**.

**Why reuse `token`:** dual-column lookup (`WHERE token = ? OR slug = ?`) doubles every read for marginal benefit. Unifying simplifies the repository, the routing (`/h/{token}` and `/{token}` work unchanged), and the cleanup job.

**Validation scope:** the slug regex applies ONLY to user-supplied input on `POST /api/endpoints`. Auto-generated tokens (`secrets.token_urlsafe(16)`, which can contain uppercase letters and underscores) bypass the validator — they are written directly to the `token` column. The `Endpoint` domain entity does not enforce the slug regex at the constructor level; that constraint lives in the use case for user-supplied slugs.

**Conflict translation:** when the user supplies a slug already taken, the `UNIQUE` constraint raises `sqlalchemy.exc.IntegrityError` from `repo.save()`. The `CreateEndpoint` use case catches this specific error and raises a new domain exception `SlugAlreadyTakenError` (added to `domain/exceptions.py`). The route handler maps that exception to `HTTPException(409)`. Repository stays free of HTTP concerns.

### Routing

No changes. `/h/{token}`, `/stream/{token}`, `/{token}`, `/api/endpoints/{token}/...` all work with vanity slugs identically to random tokens.

### Limitation: squatting

Without auth, anyone can claim `stripe` and block every future user. Acceptable for a learning project; V5 will add ownership-based claiming.

### Tests

- Unit: format validator accepts/rejects all edge cases (boundary lengths, leading/trailing hyphens, uppercase, reserved words).
- Integration: `POST /api/endpoints` with `slug=foo` → 201, then second `POST` with same slug → 409.
- Integration: `POST /api/endpoints` with reserved slug `api` → 400.
- Integration: existing test suite passes unchanged (slug omitted = V1 behavior).

## Feature 2: Search / filter captured requests

### User story

> As a developer reviewing a flood of Stripe webhook events, I want to type "payment_intent.succeeded" into a search box and see only the matching requests.

### Behavior

- `GET /api/endpoints/{token}/requests?q=<text>` filters the result set.
- Search composes with existing pagination (`limit`, `before_id`).
- Empty or missing `q` → unchanged list behavior.
- Search query > 200 chars → `400 Bad Request`.

### Indexed fields

- `method` (HTTP verb)
- `path` (URL path captured by the ingestor)
- `body_preview` (first 8 KB of body)
- `headers::text` (JSON-serialized headers — values and keys)

### Implementation

**Postgres `tsvector` generated column + GIN index.**

```sql
ALTER TABLE requests ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    to_tsvector('simple',
      coalesce(method, '') || ' ' ||
      coalesce(path, '') || ' ' ||
      coalesce(body_preview, '') || ' ' ||
      coalesce(headers::text, '')
    )
  ) STORED;

CREATE INDEX CONCURRENTLY requests_search_idx ON requests USING GIN (search_vector);
```

**Config choice: `'simple'`** (not `'english'`).
Webhook bodies are structured data (JSON, key-value pairs), not natural-language text. No stemming, no stopwords. `'payment_intent.succeeded'` and `'PAYMENT_INTENT.SUCCEEDED'` both tokenize cleanly via case-folding.

**Query function: `plainto_tsquery`** (not `to_tsquery`).
Accepts user input as a plain string, internally escapes operators. Eliminates the injection class where `to_tsquery('a & b | c')` lets users construct arbitrary expressions.

### API surface

```
GET /api/endpoints/{token}/requests
  ?q=<string, max 200 chars>
  &limit=<int, default 50, max 200>
  &before_id=<uuid, optional>
```

Response shape unchanged from V1. Search is transparent to clients.

### UI integration

- Text input above the request list in `viewer.html`.
- HTMX debounced GET on `keyup` (300ms via `hx-trigger="keyup changed delay:300ms"`), swaps the `#request-list` fragment.
- Empty input → reload full list.
- Live SSE-appended requests are NOT auto-filtered against the active query. Documented limitation; user re-runs query to refresh.

### Limitations

- **8 KB body cap.** Bodies offloaded to GCS (> 8 KB) aren't searchable. Documented in UI tooltip.
- **AND semantics, no phrase queries.** `plainto_tsquery('foo bar')` produces `'foo' & 'bar'` (rows must contain BOTH words, in any order). Exact-phrase matching ("foo bar" as adjacent words) is not supported — would require `phraseto_tsquery` and stricter positional indexing. Acceptable for V2.5.
- **Hyphenated tokens split.** `'simple'` config tokenizes on `-`, so `x-stripe-signature` is indexed as three tokens `x`, `stripe`, `signature`. Searches for the full header name find matches via AND, not exact match.
- **Headers indexed as JSONB text.** The full `headers::text` rendering (`{"content-type": "application/json"}`) is fed into `to_tsvector`. Braces/quotes/colons are stripped by `'simple'` tokenization, so both keys and values become searchable as bare words.

### Tests

- Integration: insert 3 requests with distinct body content → `?q=foo` returns only matching.
- Integration: empty `q` returns full list (V1 behavior).
- Integration: pagination + search compose (request 100 items, search returns 30, paginate with `before_id`).
- Integration: query > 200 chars → 400.

## Feature 3: Export captured requests as JSON

### User story

> As a developer building a webhook replay test fixture, I want to download all captured requests for an endpoint as a single JSON file I can feed into pytest.

### Endpoint

`GET /api/endpoints/{token}/export.json`

- `Content-Type: application/json`
- `Content-Disposition: attachment; filename="webhook-inspector-<token>-<YYYYMMDD>.json"`
- Body streamed via `StreamingResponse`.

### Response format

```json
{
  "endpoint": {
    "token": "stripe-test",
    "created_at": "2026-05-13T10:00:00Z",
    "expires_at": "2026-05-20T10:00:00Z",
    "response": {
      "status_code": 200,
      "body": "{\"ok\":true}",
      "headers": {},
      "delay_ms": 0
    }
  },
  "exported_at": "2026-05-13T12:34:56Z",
  "exported_request_count": 142,
  "requests": [
    {
      "id": "01HZJ...",
      "method": "POST",
      "path": "/",
      "headers": {"content-type": "application/json", "x-stripe-signature": "..."},
      "body": "...full body, inlined from DB or fetched from GCS...",
      "body_size": 1234,
      "received_at": "2026-05-13T12:00:00Z"
    }
  ]
}
```

- `requests[].body` is the **full** body — inlined from `body_preview` when small (≤ 8 KB) or fetched from GCS via `blob_key` otherwise. This is the main value-add over the existing API which only returns the preview.
- `exported_request_count` is the number of rows in the `requests` array, NOT the endpoint's lifetime counter (which is `endpoints.request_count` and may be higher if expired requests were cleaned up).
- Order: most recent first (matches viewer UI).

### Implementation

**Streaming generator (illustrative shape — implementation detail in the plan):**

```python
async def export_stream() -> AsyncIterator[bytes]:
    yield b'{"endpoint":'
    yield json.dumps(endpoint_dict).encode()
    yield b',"exported_at":"' + datetime.now(UTC).isoformat().encode() + b'"'
    yield b',"exported_request_count":' + str(count).encode()
    yield b',"requests":['
    first = True
    async for row in session.stream(query.execution_options(yield_per=100)):
        if not first:
            yield b','
        first = False
        # body_preview holds the full body when body_size <= inline_threshold (8 KB),
        # else it's NULL and the body lives in GCS at blob_key.
        body = row.body_preview if row.blob_key is None else await blob_store.get(row.blob_key)
        yield json.dumps({**_row_to_dict(row), "body": body}).encode()
    yield b']}'
```

- `AsyncSession.stream()` with `yield_per=100` keeps memory bounded.
- GCS bodies fetched **sequentially** during streaming. Slow for blob-heavy endpoints; acceptable for V2.5. Concurrent prefetch is a V3 optimization.
- `datetime.now(UTC)` not `datetime.utcnow()` (deprecated).

### Hard cap

- Max 10,000 requests per export, configurable via env `WEBHOOK_INSPECTOR_EXPORT_MAX_REQUESTS` (default `10000`).
- Beyond cap: respond `413 Payload Too Large` with body `{"detail":"Export exceeds 10000 requests. Filter then export will land in V3."}`.

### UI integration

Plain `<a href="/api/endpoints/{token}/export.json" download class="export-button">Export JSON</a>` in viewer header. Browser handles the download natively.

### Tests

- Integration: create endpoint, capture 3 requests (one with blob_key, two inline) → export returns valid JSON with all 3 bodies inlined.
- Integration: endpoint with 10001 captured requests → 413.
- Integration: streaming behavior — assert `Content-Type: application/json` header, body is valid parseable JSON.
- Integration: `Content-Disposition` filename includes token and date.

## Migration 0003

```sql
-- Add tsvector column with GIN index on requests table
ALTER TABLE requests ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    to_tsvector('simple',
      coalesce(method, '') || ' ' ||
      coalesce(path, '') || ' ' ||
      coalesce(body_preview, '') || ' ' ||
      coalesce(headers::text, '')
    )
  ) STORED;

CREATE INDEX CONCURRENTLY IF NOT EXISTS requests_search_idx
  ON requests USING GIN (search_vector);
```

**No changes to `endpoints` table.** Slug reuses the existing `token` column.

**`CREATE INDEX CONCURRENTLY`:** Alembic supports this via `op.execute()` + `op.get_bind().execution_options(isolation_level="AUTOCOMMIT")`. Documented in migration comment.

**Verify at implementation time:** `GENERATED ALWAYS AS (...) STORED` requires the expression to be `IMMUTABLE`. `jsonb_out` (the implicit function behind `headers::text`) is marked `IMMUTABLE` in Postgres 12+, so this should work on our Postgres 16 target. If the migration fails with `generation expression is not immutable`, fall back to a regular column populated by a `BEFORE INSERT/UPDATE` trigger.

## Acceptance criteria

1. `make test` passes (unit + integration).
2. `make lint` passes (ruff check + format).
3. `make type` passes (mypy strict).
4. Coverage on domain + application stays ≥ 95%.
5. `POST /api/endpoints {"slug":"my-stripe-test"}` returns 201 with `token == "my-stripe-test"`.
6. `POST /api/endpoints {"slug":"my-stripe-test"}` a second time returns 409 (`SlugAlreadyTakenError`).
7. `POST /api/endpoints {"slug":"api"}` returns 400 (`ReservedSlugError`).
8. `POST /api/endpoints {"slug":"FOO"}` returns 400 (`InvalidSlugError`, uppercase rejected).
9. `POST /api/endpoints` (no slug) returns 201 with a 22-char `secrets.token_urlsafe(16)` token — V1 behavior preserved.
10. `GET /api/endpoints/{token}/requests?q=stripe` filters results to rows matching "stripe" across method/path/body/headers.
11. `GET /api/endpoints/{token}/requests?q=foo bar` returns rows containing BOTH `foo` AND `bar` (AND semantics, documented).
12. `GET /api/endpoints/{token}/export.json` returns a streaming `application/json` response, with `Content-Disposition: attachment`, parseable as valid JSON, containing full bodies (inlined + GCS-fetched).
13. `GET /api/endpoints/{token}/export.json` on endpoint with > 10000 requests returns 413.
14. Existing V1+V2 endpoint-creation and request-listing behavior unchanged when new params are omitted.

## Known limitations (documented, not bugs)

- Search misses content > 8 KB (offloaded to GCS).
- No filtering on export — exports everything up to 10k cap.
- Slug squatting unaddressed pre-V5.
- Live SSE updates don't honor active search filter.
- Export of blob-heavy endpoints is slow (sequential GCS fetches).
